CI/CD with OpenShift

Simple OpenShift Build

Transcript: This image shows a very simple CI/CD Setup. When a developer pushes code to the source code repository (Git in this example) a Jenkins Pipeline build is triggered. This can happen either through a web hook in the Source code repository or by polling from Jenkins (in case Jenkins is behind a firewall).

In this example Jenkins simply orchestrates the various OpenShift tasks. First it would trigger Build Config - which would be the same S2I build that is used without Jenkins. The source code would be built in the S2I Builder Pod which results in a container image in the integrated docker registry and a new latest tag in the Image Stream. Then Jenkins would tag the image. Jenkins can trigger a deployment by using the deployment config. Finally Jenkins could tag the image again (as ProdReady-1.1 in this example), update the deployment configuration for the production project and deploy the application.

While this works it is mostly a first step into using Jenkins. Since the OpenShift Build Configuration is used to build Jenkins doesn’t actually get to "see" the source code. Therefore Jenkins can’t examine the code for unit tests, code analysis tests or even see the built artifact (WAR,EAR, …).

CI/CD with OpenShift

Complex Jenkins Build - Part 1

Transcript:

This image shows a more mature build process using Jenkins.

When the developer pushes code a pipeline is triggered. But instead of Jenkins triggering an OpenShift build config Jenkins uses a Jenkins Slave Pod (3) to build the artifact. In our case this is a WAR file therefore we will be using a Maven Slave Pod. When that build finishes the Jenkins Workspace will contain the built WAR file. This image also shows use of Nexus as an Maven Proxy Repository to speed up the build using Maven by pulling dependencies from within the cluster rather than from the internet. It would be possible to further speed up the build by mounting a PVC into the Maven Slave Pod (replace /home/jenkins/.m2 maybe with /home/jenkins/.m2/repository as your mount point to prevent overwriting the .m2 settings that are in the pod by default): https://blog.openshift.com/decrease-maven-build-times-openshift-pipelines-using-persistent-volume-claim Since the code is built by Jenkins it is now possible to run unit tests, or code analysis using SonarQube from the Pipeline (4). Finally since Jenkins now owns the built artifact (WAR file in our case) the Pipeline can upload the built WAR file into Nexus to keep a record of built artifacts.

CI/CD with OpenShift

Complex Jenkins Build - Part 2

Transcript: Continued from previous Slide.

When all tests succeed and the artifact has been pushed to Nexus Jenkins can then trigger a binary build in OpenShift (1., which will be discussed in detail in the next module). This binary build takes the built artifact either from Jenkins or (as shown here) from Nexus (2) and just adds it to the S2I builder image - which does not need to compile any source code. Again the built image is pushed to the registry and the image stream updated with the new latest tag. (3) Deployment happens as in the simple build process. (4, 5) Finally in this example the deployment to production happens through a Blue/Green deployment strategy.

One point to make is that the production environment could be another cluster - and therefore the image would need to be made available to the other cluster. Usually through pushing/copying it to an external container registry.

Customizing Jenkins

Adding Additional Plug-ins

  • Create Git repository with plugins.txt:

    slack:2.3
sonar:2.6.1

  • Start new Jenkins S2I build to create custom image:

    oc new-build jenkins:2~<git_repository_url> --name=jenkins-custom

  • Launch new Jenkins image:

    oc new-app jenkins-persistent \
    -p NAMESPACE=CUSTOM_IMAGE_NAMESPACE \
    -p JENKINS_IMAGE_STREAM_TAG=jenkins-custom:latest \
    -p MEMORY_LIMIT=1Gi -p VOLUME_CAPACITY=4Gi

    • CUSTOM_IMAGE_NAMESPACE is project where image was built

Jenkins Slave Pods

  • Jenkins preconfigured for slave pods

  • Execute builds in separate RHOCP pod

    • In same project as Jenkins pod

    • Configured in Jenkins system configuration as Kubernetes pods

    • Do not use up Jenkins pod resources

  • Slave pods included with RHOCP:

    • Base slave pod to build custom slave pods

    • Node.js

    • Maven

    • ASP.net

  • Limits/quotas enforced

    • Maven pods usually require at least 2GB memory

    • "Error 137: Pod ran out of memory and got killed"

Customizing Jenkins Slave Pods

Adding apb Tool in Docker File
FROM registry.access.redhat.com/openshift3/jenkins-slave-maven-rhel7:v3.9
USER root
RUN yum -y install apb && \
    yum clean all
USER 1001

Transcript:

  • This example uses the Maven slave pod as the foundation.

  • The build is executed as the root user.

  • yum is used to install the apb tool and all of its dependencies.

  • Lastly the user is switched back to user 1001, which is the user the base image uses.

  • Note that there is no entry point defined—we are simply adding a tool to the Maven base image and will take everything from the base image as defined.

Nexus

settings.xml

  • Maven uses settings.xml in $HOME/.m2 for configuration outside of pom.xml:

    <?xml version="1.0"?>
<settings>
  <mirrors>
    <mirror>
      <id>Nexus</id>
      <name>Nexus Public Mirror</name>
      <url>http://nexus-opentlc-shared.apps.na39.openshift.opentlc.com/content/groups/public/</url>
      <mirrorOf>*</mirrorOf>
    </mirror>
  </mirrors>
  <servers>
  <server>
    <id>nexus</id>
    <username>admin</username>
    <password>admin123</password>
  </server>
</servers>
</settings>

SonarQube

  • Open source code-analysis tool

  • Analyze source code for common programming mistakes

  • Plug-ins for common programming languages

  • Can call via Maven or Jenkins plug-in

  • Add to pom.xml:

    <plugin>
  <groupId>org.sonarsource.scanner.maven</groupId>
  <artifactId>sonar-maven-plugin</artifactId>
  <version>3.4.0.905</version>
</plugin>

  • Call SonarQube scanner from Maven:

    mvn sonar:sonar -s ./nexus_settings.xml \
   -Dsonar.host.url=http://sonarqube-xyz-sonarqube.apps-na39.openshift.opentlc.com

Skopeo

Overview

  • Command-line utility that performs various operations on container images and image repositories

  • Works with OCI images as well as original Docker v2 images:

    • API V2 registries such as Docker registries

    • Atomic registry

    • Private registries

    • Local directories

    • Local OCI-layout directories

  • Does not require a Docker daemon to be running to perform operations

Skopeo

Operations

  • Copy image from and to various storage mechanisms

    • Example: can copy images from one registry to another without requiring privilege

  • Inspect properties of remote image, including layers, without pulling image to host

  • Delete image from image repository

  • When required by repository, skopeo can pass appropriate credentials and certificates for authentication

Skopeo

Useful Targets

  • docker://docker-reference

    • Image in registry implementing Docker Registry HTTP API V2

    • Example: docker://docker.io/xyz-jenkins/jenkins-slave-maven-appdev:v3.9

  • docker-daemon:docker-reference

    • Image docker-reference stored in Docker daemon internal storage

    • Example: docker-daemon:jenkins-slave-maven-appdev:v3.9

  • More options available: dir:path, containers-storage, docker-archive, oci:path:tag, ostree:image

Transcript:

Skopeo operates on the following image and repository types:

  • containers-storage:docker-reference An image located in a local containers/storage image store. Location and image store specified in /etc/containers/storage.conf

  • dir:path An existing local directory path storing the manifest, layer tarballs and signatures as individual files. This is a non-standardized format, primarily useful for debugging or noninvasive container inspection.

  • docker://docker-reference An image in a registry implementing the "Docker Registry HTTP API V2". By default, uses the authorization state in $HOME/.docker/config.json, which is set e.g. using (docker login).

  • docker-archive:path[:docker-reference] An image is stored in the docker save formatted file. docker-reference is only used when creating such a file, and it must not contain a digest.

  • docker-daemon:docker-reference An image docker-reference stored in the docker daemon internal storage. docker-reference must contain either a tag or a digest. Alternatively, when reading images, the format can also be docker-daemon:algo:digest (an image ID).

  • oci:path:tag An image tag in a directory compliant with "Open Container Image Layout Specification" at path.

  • ostree:image[@/absolute/repo/path] An image in local OSTree repository. /absolute/repo/path defaults to /ostree/repo.
