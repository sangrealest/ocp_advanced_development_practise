Build Definition

  • Process of transforming input parameters into resulting object

  • Builds used to transform source code into runnable container image

  • Build configuration or BuildConfig file characterized by:

    • Build strategy

    • At least one source

  • Strategy determines process; sources provide input

Source Types

  • Six source types given as build input:

    • Git

    • Dockerfile

    • Binary

    • Image

    • Input secrets

    • External artifacts

  • Binary and Git mutually exclusive source types

  • Dockerfile and image used by themselves, with each other, or together with Git or binary

  • Binary source type is unique in how specified to system

Build Configuration

Build Run

  • Working directory constructed and input content placed there

    • Example:

      • Input Git repository cloned into working directory

      • Files specified from input images copied into working directory using target path

  • Build process changes directories into contextDir (if defined)

  • Inline Dockerfile, if any, written to current directory

  • Current directory content provided to build process for reference by:

    • Dockerfile

    • assemble script

    • Custom builder logic

  • Input content residing outside contextDir ignored by build

Build Input

Dockerfile Source

  • When BuildConfig.spec.source.type is Dockerfile:

    • Inline Dockerfile used as build input

    • No additional sources can be provided

  • Source type valid when build strategy Docker or Custom type

  • Source definition part of BuildConfig spec section:

    source:
  type: "Dockerfile"
  dockerfile: "FROM centos:7\nRUN yum install -y httpd" (1)
    1dockerfile field contains inline Dockerfile to be built

Build Input

Image Source

  • Additional files can be provided to build process via images

  • Input images referenced same way that From and To image targets are defined

    • Can reference both container images and image stream tags

  • To copy image and destination to place them in build context:

    • Must provide at least one path pair to indicate path of files or directories

    • Source path can be absolute path within specified image

    • Destination must be relative directory path

  • Image inputs specified in source definition of BuildConfig

Build Input

Git Source

  • BuildConfig.spec.source.type is Git

  • Git repository required

  • Inline Dockerfile optional

  • Source code fetched from location specified

  • If BuildConfig.spec.source.dockerfile field specified:

    • Inline Dockerfile replaces one in contextDir of Git repository

  • Source definition part of spec section in BuildConfig

Copyright ©2012-2017 Red Hat, Inc. - 9d795b8f3f6f1145ec9979c95ebfe7065236f94f

Build Input

Git Source Example
source:
  type: "Git"
  git: (1)
    uri: "https://github.com/openshift/ruby-hello-world"
    ref: "master"
  contextDir: "app/dir" (2)
  dockerfile: "FROM openshift/ruby-22-centos7\nUSER example" (3)
1git field contains URI to remote Git repository of source code. Optionally, specify ref field to check out specific Git reference. Valid ref can be SHA1 tag or branch name.
2contextDir field allows default location override inside source code repository where build looks for application source code. If application exists inside subdirectory, can override default location (root folder).
3If optional dockerfile field provided, should be string containing Dockerfile that overwrites any Dockerfile in source repository.

Build Input

Binary Source Type Build

  • Streaming content in binary format from local file system to builder

  • Unique—leveraged solely based on use of oc start-build

    Binary type builds require content to be streamed from the local file system, so automatically triggering a binary type build (for example, via an image change trigger) is not possible, because the binary files cannot be provided. Similarly, you cannot launch binary type builds from the web console.

Build Input

Starting Build with Binary Source

  • Invoke oc start-build with one of following options:

--from-file

  • Specified file contents sent as binary stream to builder

  • Builder stores data in file with same name at top of build context

--from-dir

and

--from-repo

  • Contents archived, sent as binary stream to builder

  • Builder extracts archive’s contents in build context directory

--from-archive

  • Specified archive sent to builder, where extracted in build context directory

  • Option behaves same as --from-dir

  • Archive created on host first, whenever argument to these options is directory

Build Input

Using External Artifacts with S2I

  • Storing binary files in source repository not recommended

  • Instead, define build to pull additional files during build process

    • Java JAR dependencies

  • How this is done depends on build strategy

  • For Source build strategy, put proper shell commands into assemble script:

    • .s2i/bin/assemble file:

      #!/bin/sh
APP_VERSION=1.0
wget http://repository.example.com/app/app-$APP_VERSION.jar -O app.jar

  • To create run script to run application:

    • .s2i/bin/run file:

      #!/bin/sh
exec java -jar app.jar

Build Input

Using External Artifacts with Docker Strategy

  • Modify Dockerfile and invoke shell commands with RUN instruction:

    FROM jboss/base-jdk:8

ENV APP_VERSION 1.0
RUN wget http://repository.example.com/app/app-$APP_VERSION.jar -O app.jar

EXPOSE 8080
CMD [ "java", "-jar", "app.jar" ]
In practice, you may want to use an environment variable for the file location so that the specific file to be downloaded can be customized using an environment variable defined in BuildConfig, rather than updating the assemble script or Dockerfile.

Build Secrets

Source Clone Secrets

  • Builder pods require access to Git repositories defined as build source

  • Source clone secrets provide builder pod access to:

    • Private repositories

    • Repositories with self-signed or untrusted SSL certificates

  • Supported source clone secret configurations:

    • .gitconfig file

    • Basic authentication

    • SSH key authentication

    • Trusted certificate authorities

      You can use combinations of configurations as needed.

Build Secrets

.gitconfig Secret

  • If application cloning depends on .gitconfig file:

    • Create secret containing it

    • Add to builder service account

    • Add to BuildConfig

  • To create secret from .gitconfig file:

    oc create secret generic mysecret --from-file=.gitconfig=<path/to/.gitconfig>

SSL verification can be turned off if sslVerify=false is set for the http section in the .gitconfig file:

[http]
  sslVerify=false

Build Secrets

SSH Key Authentication

  • Requires private SSH key

  • Repository keys located in $HOME/.ssh/ directory

  • By default, uses keys named id_dsa.pub, id_ecdsa.pub, id_ed25519.pub, id_rsa.pub

  • Create secret before using SSH key to access private repository

  • Upload public key to SCM—used to access your private repository

  • To generate SSH key credentials:

    $ ssh-keygen -t rsa -C "your_email@example.com"
    Creating a passphrase for the SSH key prevents OpenShift from building. When prompted for a passphrase, leave it blank.
Copyright ©2012-2017 Red Hat, Inc. - 9d795b8f3f6f1145ec9979c95ebfe7065236f94f

Build Secrets

SSH Key Authentication

SSH Authentication Type

oc Command Syntax

To create ssh secret:

oc create secret generic sshsecret \
    --from-file=ssh-privatekey=$HOME/.ssh/id_rsa

To create SSH-based authentication secret with .gitconfig file:

oc create secret generic sshsecret               \
    --from-file=ssh-privatekey=$HOME/.ssh/id_rsa \
    --from-file=.gitconfig=</path/to/file>

To create secret that combines .gitconfig file and CA certificate:

oc create secret generic sshsecret           \
    --from-file=ca.crt=<path/to/certificate> \
    --from-file=.gitconfig=</path/to/file>

To create basic authentication secret with CA certificate file:

oc create secret generic <secret_name>  \
    --from-literal=username=<user_name> \
    --from-literal=password=<password>  \
    --from-file=ca.crt=<path/to/certificate>

To create basic authentication secret with .gitconfig file and CA certificate file:

oc create secret generic <secret_name>     \
    --from-literal=username=<user_name>    \
    --from-literal=password=<password>     \
    --from-file=.gitconfig=</path/to/file> \
    --from-file=ca.crt=<path/to/certificate>

Input Secrets

  • Node.js example also applicable when adding:

    • SSL certificates into /etc/ssl/certs directory

    • API keys or tokens

    • Other secrets

Build Secrets

Creating Input Secrets

  • Add input secret to an existing BuildConfig

  • To create secret:

    oc create secret generic secret-npmrc \
   --from-file=.npmrc=~/.npmrc

    • Command creates secret-npmrc containing Base64-encoded content of ~/.npmrc file

Adding Input Secrets

  • To include secret in new BuildConfig:

    $ oc new-build \
    openshift/nodejs-010-centos7~https://github.com/openshift/nodejs-ex.git \
    --build-secret secret-npmrc

  • During build, .npmrc file copied into directory where source code located

  • To specify destination directory when creating BuildConfig:

    $ oc new-build \
    openshift/nodejs-010-centos7~https://github.com/openshift/nodejs-ex.git \
    --build-secret “secret-npmrc:/etc”

S2I Force Pull

  • Used if builder image specified in build configuration available locally on node

  type: "Source"
  sourceStrategy:
    from:
      kind: "ImageStreamTag"
      name: "builder-image:latest" (1)
    forcePull: true (2)
    1Specify builder image name to override local image
    2Set forcePull tag to true to refresh image from image stream registry

Build Strategy

S2I Incremental Builds

  • Reuse artifacts from previously built images

  • To create an incremental build, modify BuildConfig strategy definition:

    strategy:
  type: "Source"
  sourceStrategy:
    from:
      kind: "ImageStreamTag"
      name: "incremental-image:latest" (1)
    incremental: true (2)
    1Specify image that supports incremental builds
    2Flag controls whether incremental build attempted

  • Consult builder image documentation to determine if behavior supported

  • If builder image does not support incremental builds, build still succeeds

  • Log message states incremental build not successful because of missing save-artifacts script

Build Strategy

Overriding Builder Image Scripts

  • Provide assemble, run, save-artifacts S2I scripts in s2i/bin directory of application source repository

  • If script found, used in place of same named script provided in image

  • Provide URL of directory containing scripts as part of strategy definition:

    strategy:
  type: "Source"
  sourceStrategy:
    from:
      kind: "ImageStreamTag"
      name: "builder-image:latest"
    scripts: "http://somehost.com/scripts_directory" (1)
    1run, assemble, and save-artifacts appended to path

Making Environment Variables Available to Builds

  • Environment files:

    • .s2i/environment file in source repository

    • Environment variables specified in file present during build process and in final container image

  • BuildConfig environment:

    • Add environment variables to sourceStrategy definition of BuildConfig

    • Environment variables defined as visible during assemble script execution and defined in output image, making them available to run script and application code

  • Examples:

    • DISABLE_ASSET_COMPILATION=true

Advanced Build Operations

Setting Build Resources

  • By default, builds completed by pods using unbound resources

  • resources, cpu, memory parameters optional

  • Specify resource limits in project’s default container limits

  • Specify resource limits as part of build configuration:

    apiVersion: "v1"
kind: "BuildConfig"
metadata:
  name: "sample-build"
spec:
  resources:
    limits:
      cpu: "100m" (1)
      memory: "256Mi" (2)
    1cpu in CPU units: 100m represents 0.1 CPU units (100 * 1e-3)
    2memory in bytes: 256Mi represents 268,435,456 bytes (256 * 2 ^ 20)

Advanced Build Operations

Setting Build Resources with Quotas in Place

  • If quota defined for project, resources section or limit range required:

    • Limit range defined in project, where defaults from LimitRange object apply to pods created during build process

    • resources section set with explicit requests:

      resources:
  requests: (1)
    cpu: "100m"
    memory: "256Mi"
      1requests object contains resource list corresponding to resource list in quota

  • Otherwise, creating build pod fails, citing failure to satisfy quota

Advanced Build Operations

Defining Maximum Duration

  • Set completionDeadlineSeconds field

    • Specified in seconds

    • Not set by default

    • When not set, no maximum duration enforced

  • Maximum duration counted from time build pod scheduled

    • Defines active duration time, including time to pull builder image

  • Build terminated by OpenShift after specified timeout reached

  • BuildConfig specifying completionDeadlineSeconds field for 30 minutes:

    spec:
  completionDeadlineSeconds: 1800

Advanced Build Operations

Assigning Builds to Specific Nodes

  • Builds can be targeted to run on specific nodes by specifying labels in nodeSelector field of build configuration

  • nodeSelector value set of key-value pairs matched to node labels when scheduling build pod:

    apiVersion: "v1"
kind: "BuildConfig"
metadata:
  name: "sample-build"
spec:
  nodeSelector:(1)
    region: primary
    nodetype: build
    1Builds associated with build configuration run only on nodes with region=primary and nodetype=build labels

  • nodeSelector value can be controlled by cluster-wide default and override values

  • Defaults applied only if build configuration:

    • Does not define any key-value pairs for nodeSelector

    • Also does not define explicitly empty map value of nodeSelector:{}

  • Override values replace values in build configuration on key-by-key basis

Advanced Build Operations

Chaining Builds

  • For compiled languages Go, C, C++, and Java

  • Includes dependencies necessary for compilation in application image:

    • May increase image size

    • Introduces vulnerabilities that can be exploited

  • Chain two builds to avoid issues:

    • Build 1 produces compiled artifact

    • Build 2 places artifact in separate image that runs artifact

  • Compiles artifact and places in separate runtime image

Advanced Build Operations

Chaining Builds BuildConfig - Build 1

  • First build produces image containing WAR

  • Image pushed to artifact-image image stream

  • Output artifact path depends on assemble script of S2I builder

  • Output artifact /wildfly/standalone/deployments/ROOT.war:

    apiVersion: v1
kind: BuildConfig
metadata:
  name: artifact-build
spec:
  output:
    to:
      kind: ImageStreamTag
      name: artifact-image:latest
  source:
    git:
      uri: https://github.com/openshift/openshift-jee-sample.git
    type: Git
  strategy:
    sourceStrategy:
      from:
        kind: ImageStreamTag
        name: wildfly:10.1
        namespace: openshift
    type: Source

Advanced Build Operations

Chaining Builds BuildConfig - Build 2

  • Uses image source with path to WAR file inside output image from first build

  • Inline Dockerfile copies WAR file into runtime image:

    apiVersion: v1
kind: BuildConfig
metadata:
  name: image-build
spec:
  output:
    to:
      kind: ImageStreamTag
      name: image-build:latest
  source:
    type: Dockerfile
    dockerfile: |-
      FROM jee-runtime:latest
      COPY ROOT.war /deployments/ROOT.war
    images:
    - from: (1)
        kind: ImageStreamTag
        name: artifact-image:latest
      paths: (2)
      - sourcePath: /wildfly/standalone/deployments/ROOT.war
        destinationDir: "."
  strategy:
    dockerStrategy:
      from: (3)
        kind: ImageStreamTag
        name: jee-runtime:latest
    type: Docker
  triggers:
  - imageChange: {}
    type: ImageChange (4)
    1from specifies that Docker build includes output of image from artifact-image image stream, target of previous build
    2paths specifies which paths from target image to include in current Docker build
    3Runtime image used as source image for Docker build
    4ImageChange trigger invokes second build every time first build finishes successfully
